# План впровадження тестів для GUI та Mock-об'єктів для SSH

Цей план описує кроки для додавання тестування графічного інтерфейсу та створення механізму імітації (mocking) відповідей SSH-сервера для забезпечення надійності проєкту згідно з принципами TDD.

## 1. Рефакторинг SSH-клієнта (Модуль `ssh_client.rs`)
Для можливості імітації з'єднання необхідно абстрагувати роботу з SSH через trait.

- [ ] Визначити trait `SSHConnector` з методами `connect` (опціонально як фабрика) та `execute`.
- [ ] реалізувати `SSHConnector` для `SSHClient`.
- [ ] Створити `MockSSHClient`, який дозволяє заздалегідь задавати відповіді на команди.
- [ ] Додати unit-тести для перевірки логіки самого `MockSSHClient`.

## 2. Адаптація додатку (`app.rs`)
Додаток має працювати з абстракцією, а не з конкретною реалізацією.

- [ ] Змінити поле `client` у структурі `WhitelistApp` на `Option<Box<dyn SSHConnector>>` (або аналогічну абстракцію).
- [ ] Забезпечити правильну обробку поля `client` при серіалізації/десеріалізації (використовуючи `#[serde(skip)]`).
- [ ] Додати метод для ін'єкції mock-клієнта під час тестів.

## 3. Впровадження GUI-тестів
Оскільки `egui` працює в режимі негайного малювання (immediate mode), ми будемо тестувати логіку зміни стану додатку.

- [ ] Додати блок `#[cfg(test)] mod tests` у файл `app.rs`.
- [ ] Написати тести для:
    - Процесу підключення та обробки помилок.
    - Оновлення списку адрес при отриманні даних від mock-клієнта.
    - Логіки переходу між вкладками.
    - Скидання стану при помилках зв'язку.

## 4. Створення Mock-даних
- [ ] Створити набір констант або окремих файлів з прикладами виводу MikroTik (leases, networks, servers) для використання в тестах.
- [ ] Реалізувати можливість "програмування" `MockSSHClient` на послідовність різних відповідей для тестування складних сценаріїв (наприклад, успішний запит після помилки).

## 5. Перевірка та TDD
- [ ] Запустити `cargo test` для перевірки всіх існуючих та нових тестів.
- [ ] Використовувати цикл Red-Green-Refactor для кожної нової функції тестування.
